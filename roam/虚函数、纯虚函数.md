# 虚函数

**what**：
virtual关键字修饰的成员函数，允许派生类重写基类的同名函数，实现多态（运行时）。具体来说：通过基类的 指针或引用 调用虚函数时会调用 基类指针或引用 指向或引用的类对象的虚函数版本
> **底层**：
> - 有虚函数的类（`virtual`）有 虚函数表、虚函数指针（ `*__vptr`），其派生类也有自己的虚函数表。
> - 在类对象初始化的时候，虚函数指针 指向对应的虚函数表（eg：派生类创建时，虚函数表指针指向派生类的虚函数表，表内会用掉重写版本的函数指针替换被重写的）。
> - 调用虚函数时，根据类指针 -> 找到 `*__vptr`-> 再找到对应的 虚函数表 -> 调用可调用的函数版本。
>
> **性能**：虚函数 比 非虚函数 *调用慢* 、 *体积大*
>  
>> **虚函数表**：一个数组，里面存放函数指针（派生类有重写存放重写版本，否则存放基类版本），指向该类可以调用的函数版本。 *编译时生成* -> 有虚函数的类及其派生类各自都有
>
>>**虚函数指针**：指向虚函数表。 *运行时创建对象时确定* ->  创建类对象时，在基类添加，然后指向该类（基类或派生类）的虚表

```cpp
class Base {
public:
    // VirtualTable* __vptr; 编译器自动添加 虚函数指针
    virtual void function1() {};
    virtual void function2() {};};

// D1、D2 是派生类，会继承 虚函数指针
class D1: public Base {
public:
    void function1() override {};};

class D2: public Base {
public:
    void function2() override {};};

int main() {
  D1 d1 {};           // D1创建，*__vptr 指向 D1虚表
  Base* dPtr = &d1;   // Base指针 指向 D1的基类部分，又 *__vptr 指向 D1虚表，因此可以访问 D1的虚函数表
  dPtr->function1();  // 程序识别出这是虚函数，通过 虚函数表 解析出 D1::function1()
  Base b {};          // Base创建，*__vptr 指向 Base虚表
  Base* bPtr = &b;
  bPtr->function1();  // 调用 Base::function1
}
```

**why**：
1. **更灵活**：通过父类指针调用函数，在运行时确定调用版本；没有虚函数则只能调用父类版本
2. **可扩展**：无需修改原有逻辑，新增子类就能扩展功能[^1]


## 构造函数/析构函数 中能否调用 虚函数？

可以，但不建议，因为可能没法触发多态，无法达到我们的目的
- 构造派生类，先构造基类，基类构造函数中调用虚函数，希望触发派生类版本，但派生类还没构造，只能触发基类版本
- 析构派生类，先析构派生类，后析构基类，基类中调用虚函数，希望触发派生类版本，但派生类已经析构了，只能触发基类版本  


# 纯虚函数

**what**：
虚函数末尾添加`=0`标记的函数

**why**：
1. **定义接口规范（接口类）**：派生类必须实现所有纯虚函数（让派生类实现我们想要的功能）
2. **防止抽象概念实例化（抽象类）**[^2]

> **接口类**：仅包含纯虚函数的类。
> 1. 没有成员变量，所有函数都是纯虚函数
> 2. 接口类可作为函数参数使用  
> 
> **why**：
> 定义接口规范， *解耦* 接口（做什么）和实现（怎么做）

> **抽象类**：未实现基类的所有纯虚函数的派生类
> 1. 抽象类无法实例化（表示是没实现要求的接口规范，没法用）
> 2. 抽象类有虚函数表
> 
> **why**：
> 防止抽象概念实例化


[^1]: 比如一个图形库，基类是 Shape，有个 draw 虚函数。一开始只有 Circle 和 Square 子类，调用 draw 时分别画圆和正方形。后来要加 Triangle 类，只要让它继承 Shape 并重写 draw 函数。原来遍历 Shape 指针数组画图的代码完全不用改，直接新增 Triangle 对象就能正常工作

[^2]: 像 “形状”“动物” 这类抽象概念本身不应该被实例化，纯虚函数让类成为抽象类，从语法上禁止创建对象，保证设计合理性。
