# 1 HTTPS
**what：**
HTTPS 在 HTTP 基础上增加了 TLS/SSL 协议的安全应用层协议
1. 更安全，但更慢（TCP三次握手 后还需要 TLS四次握手）
2. 端口： 443
3. 增加了SSL/TLS层：提供
   - 加密传输：防窃听（通信建立前使用非对称加密，建立后使用对称加密）
   - 校验机制：防篡改（摘要算法）
   - 数字证书：验证服务器身份，防冒充

## 1.2 对称加密、非对称加密
### 1.2.1 对称加密
**what：**
加密解密使用同一把密钥
1. 加密解密的速度快
2. 安全性较低（在网络中传递密钥，容易被偷，导致数据安全无法得到保障）

```
你给我密钥，我把密钥加密后的数据给你，中间人偷到密钥，中间人能解密
```

### 1.2.2 非对称加密
**what：**
加密解密使用一对密钥，公钥可以公开传播，私钥必须保密（公钥/私钥 加密，私钥/公钥 解密）
1. 加密解密的速度慢
2. 安全性高（在网络中传递公钥，安全）

```
你给我公钥，我把公钥加密后的数据给你，中间人偷到公钥，中间人无法解密
```

## 1.3 服务端申请数字证书
![[Pasted image 20260519162241.png]]
1. 服务端将公钥和自己相关的信息交给 中间CA，中间CA 用这些信息计算出hash值，再用自己的私钥加密，生成 数字签名，
2. 中间CA 向 根CA 申请数字证书同理：根CA 计算 中间CA 数字证书的哈希值，用自己的私钥加密，得到 中间CA 数字证书的数字签名
3. 根CA 也用自己的私钥自签名，将自己的数字证书 安装到客户端的浏览器/操作系统中

### 1.3.1 数字证书
**what：**
由 CA 签发的、用于验证通信实体身份和公钥合法性的电子文档（ 数字证书 = 数字签名 + 公钥A + 服务端相关信息 ），可以形成 证书链，如下图

![[Pasted image 20260519163032.png]]

### 1.3.2 数字签名
**what：**
证明数字证书的有效性

### 1.3.3 CA
**what：**
证书颁发机构

#### 为什么需要CA？
```
背景：你发布公钥A，先给到中间人，中间人不给我，中间人将自己的公钥B给我，我用公钥B加密后，中间人用它的私钥B解密出真实内容，又用公钥A加密后给你，你用私钥A解密发现正确，于是建立了：我-中间人-你 这样的连接

因此我们需要第三方解决公钥信任问题，也就是 证书颁发机构（CA）
你把公钥放到大家都信任的CA那里，CA根据公钥和其他信息生成 数字证书，你把数字证书给我，我发现是信任的CA，于是我取出公钥...
```

#### 为什么根CA不直接颁发证书？
```
假设只有根CA，根CA失守了，整个信任链都会有问题；
有中间CA的情况，中间CA失守，问题不大
```
