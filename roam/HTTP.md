#计算机网络
# 1 HTTP
**what：**
基于 请求-响应模型 的 无状态 应用层协议，用于在互联网上传输超文本
1. 半双工
2. 端口：80
3. 明文传输，简单快速，存在安全漏洞
   - 窃听
   - 篡改
   - 冒充

> **无状态**：每个请求独立，服务器不会保留之前请求的状态信息，可通过 Cookie 实现状态管理

## 1.2 HTTP 请求方法
**what：**
HTTP 定义了 客户端和服务端之间的通信方式；HTTP请求方法 表示 对资源执行的操作

| 请求方式   | 语义            | 说明                             |
| ------ | ------------- | ------------------------------ |
| GET    | 获取数据          | 幂等；<br>数据以键值对形式存在 URL 中，安全性不好  |
| POST   | 提交数据，常用于创建新资源 | 非幂等；<br>数据以 JSON 形式存在请求体中，安全性好 |
| PUT    | 更新数据          | 幂等                             |
| DELETE | 删除数据          | 幂等                             |

> **幂等**：多次请求，结果相同
> **非幂等**：多次请求，结果不同

## 1.3 HTTP 状态码

|分类|状态码|作用|
|---|---|---|
|1xx（提示信息）|—|—|
|2xx（成功）|200 OK|请求成功，响应头包含数据|
||201 Created|请求成功，在服务器上创建了新的资源|
||204 No Content|请求成功，响应头没有数据|
|3xx（重定向）|301 Moved Permanently|永久重定向，资源不在，用 新 URL 访问|
||302 Found|临时重定向，资源在，临时用 新 URL 访问|
||304 Not Modified|资源未修改（告诉客户端可以使用缓存资源）|
|4xx（客户端错误）|400 Bad Request|请求报文有误|
||403 Forbidden|服务器禁止访问资源|
||404 Not Found|在服务器上没有找到资源|
|5xx（服务端错误）|500 Internal Server Error|通用错误（具体未知）|
||501 Not Implemented|功能未实现（客户端请求的）|
||502 Bad Gateway|服务器作为网关 / 代理，访问后端服务器错误|
||503 Service Unavailable|服务器很忙，无法响应客户端|

## 1.4 HTTP 常见字段
**客户端**
- Host：指定 要访问的服务器的域名
- Connection：客户端要求服务端使用 HTTP长连接 机制（HTTP/1.1 默认长连接，老版本 HTTP 需要指定 Connection: Keep-Alive）
- Accept：接受的数据格式
- Accept-Encoding：接受的数据压缩方式

**服务端**
- Content-Length：响应的数据长度
- Content-Type：响应的数据格式
- Content-Encoding：响应的数据压缩方法
- Cache-Control：相对时间（资源在客户端的有效期），优先级更高
- Expires：绝对时间（资源在客户端的有效期）

## 1.5 HTTP 缓存机制
### 1.5.1 强缓存
**what：**
浏览器自己判断是否使用资源
1. 浏览器第一次请求资源时，服务器返回 资源 + Cache-Control/Expires字段（说明资源的过期时间），
2. 浏览器后续请求相同资源时，会与 Cache-Control 进行比对，决定用缓存/重新请求服务器
   - Cache-Control：资源过期 相对时间，优先级更高
   - Expires：资源过程 绝对时间

### 1.5.2 协商缓存
**what：**
浏览器和服务器协商后，判断是否使用资源
1. 浏览器第一次请求资源时，服务器返回 资源 + 资源标识（Last-Modified / ETag）
2. 浏览器后续请求（ If-Modified-Since / If-None-Match ），服务器判断 若是最新资源，返回 304 状态码 ，直接从缓存拿资源；否则返回 200 状态码 + 最新资源 + 新的资源标识
   - (请求头) If-Modified-Since：用服务器发来的 Last-Modified 的值填充
    (响应头) Last-Modified：资源上一次修改时间
   - (请求头) If-None-Match：用服务器发来的 Etag 的值填充
    (响应头) Etag：资源对应的唯一字符串

> 优先使用 Etag
> - Last-Modified 的值 只精确到秒级 -> 资源修改速度在秒级以下，则无法及时更新资源
> - 如果每隔一段时间重复生成 内容相同 的文件 -> Last-Modified 每次都会返回

## 1.6 HTTP/1.0、HTTP/1.1、HTTP/2.0、HTTP/3.0
1. HTTP/1.0 是 短连接，每次 请求-响应 需要 TCP三次握手和四次挥手，效率低下

2. HTTP/1.1 引入 长连接、管道 -> 解决了 请求队头阻塞，没解决响应队头阻塞问题，提高了性能
   - 长连接：任意一方没有提出断开连接，则保持TCP连接状态（当然，如果某个HTTP长连接超过一定时间没有交互，则服务端主动断开） Connection字段 改为 Keep-Alive
   - 管道：解决了 请求队头阻塞 ，具体来说：客户端发送请求后，无需等待服务器响应，可以连续的发送请求 -> 减少整体响应时间；但 未解决 响应队头阻塞（按顺序响应）

3. HTTP/2.0 基于 HTTPS -> 更安全
   引入 头部压缩、二进制格式、多路复用、服务器主动推送 机制 -> 大幅提高性能，解决 HTTP 队头阻塞问题，但没解决 TCP层面的队头阻塞问题
   - 头部压缩：使用 HPack算法，在客户端和服务端同时维护一张 头信息表 ，所有字段存入该表，并生成对应索引号，后续只发索引号
   - 二进制格式：请求头、请求体 都是 二进制格式，被称为 frame，这样计算机就不用将明文转成二进制，再进行解析了，而是可以直接解析
   - 多路复用：一个连接中多个“流”，每个请求-响应 对应 一个流，流之间不阻塞 -> 解决HTTP队头阻塞问题（不再需要按序收发）
   - 服务器主动推送：在客户端没明确请求时，主动推送客户端可能需要的资源（eg：客户端请求 /index.html，服务器主动推送 /style.css、/logo.png 文件）

4. HTTP/3.0 使用基于 UDP 的 QUIC协议 -> 实现类似TCP的可靠传输，解决了TCP层面的队头阻塞

> **TCP队头阻塞**：TCP层必须保证收到的字节数据是完整且连续的，否则无法从内核缓冲区返回给应用层的HTTP应用。假设每次内核缓冲区读满才交给应用层的HTTP应用，但是中间有个TCP报文丢失，必须等收到了这个TCP报文才能交给HTTP应用
